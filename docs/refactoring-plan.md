# Refactoring Plan — locationjoystick

Generated by thermo-nuclear code quality review.

---

## Critical Findings

### 1. `SettingsScreen.kt` — 26-parameter composable signature + duplicated import dialogs

**File:** `feature/settings/impl/...SettingsScreen.kt:303`

26 individual lambdas instead of a `(SettingsAction) -> Unit` single callback — the pattern `MapScreen` already uses. Every new setting adds another parameter to the signature, the preview stub, and every call site. The four `ImportConfirmDialog` blocks (lines 141–199) managing four separate pending-URI state variables should collapse into a single `PendingImport` sealed class with one dialog.

**Fix:** Replace 26-lambda signature with `onAction: (SettingsAction) -> Unit`. Introduce `sealed class PendingImport` to collapse the 4 import dialog variants.

---

### 2. `SettingsViewModel.kt` — Context in ViewModel, triplicated state, nested combine pyramid

**File:** `feature/settings/impl/...SettingsViewModel.kt`

Three structural problems compounding each other:

- **Context I/O in ViewModel:** Four public functions receive `Context` and open file streams (`writeExportToUri`, `importSettings(context,uri)`, `importFromGpsJoystick`, `importFromYamla`). This is untestable.
- **Triplicated state:** `DraftState`, `RepoState`, and `SettingsUiState` all carry the same 20 fields. Adding a setting requires touching 4 places.
- **Nested combine pyramid** (lines 151–243): Root cause is that `SettingsRepository` exposes 20+ individual `Flow<T>` scalars. If the repo exposed `Flow<AppSettings>`, the pyramid disappears.
- **QR chunk session** in a `mutableMapOf` with no synchronization and no expiry (lines 62–67).

**Fix:** (a) Introduce `ImportExportRepository` for file I/O. (b) Unify state into a single `SettingsUiState` with a diff/draft mechanism. (c) Add `Flow<AppSettings>` to `SettingsRepository`.

---

### 3. `AppPreferencesDataSource.kt` — 26× copy-paste read pattern

**File:** `core/datastore/...AppPreferencesDataSource.kt:251–696`

The same 5-line `dataStore.data.catch { ... }.map { prefs[key] ?: default }` block appears **26 times**. A single private helper would eliminate ~130 lines (16% of the file). Additionally: a parallel constants hierarchy in the companion re-aliases everything from `AppConstants` for no reason; the `RecentSearch` serialization uses hand-rolled `org.json.JSONObject` instead of `kotlinx-serialization`.

**Fix:** Extract `private fun <T> pref(key, default): Flow<T>` helper. Delete companion constant aliases. Replace hand-rolled JSON with `@Serializable`.

---

### 4. `MockLocationService.kt` — God service with 17 `@Volatile` fields + asymmetric cleanup

**File:** `core/location/...MockLocationService.kt`

758-line service owns: foreground lifecycle, test provider management, position update loop, 10-action intent routing, realism snapshot, satellite count state, overlay service startup, and elevation sensor injection. Also:

- 17 `@Volatile` fields for related state that is never co-updated atomically.
- Waypoint string parsing inline in `onStartCommand` (should live in `MockLocationIntentBuilder`).
- `stopSpoofing()` calls `stopSelf()` but `onDestroy()` has a different cleanup path — last location not persisted if process is killed externally.
- Overlay services started via stringly-typed class names that won't fail at compile time on rename.

**Fix:** Extract `RealisticsEngine`, `SatelliteCountChurn`, `TestProviderManager` as collaborators. Move waypoint decoding to `MockLocationIntentBuilder`. Fix `onDestroy` to persist last location. Use typed class references for overlay services.

---

### 5. `MapViewModel.kt` — 450-line `when` block + dual data-source injection + unguarded mutable field

**File:** `feature/map/impl/...MapViewModel.kt`

- 36-branch `onAction` `when` expression (lines 226–577) is untestable as a monolith.
- `PreferencesDataSource` injected *alongside* `SettingsRepository` — two data access paths for the same store (line 83). The direct `preferencesDataSource.getRoamingDefaults()` call at line 184 bypasses the repository layer.
- `latestRoamingDefaults: RoamingDefaults` is a plain `var` written from a coroutine — no `@Volatile`, no `StateFlow` — read at `OpenRoamingSheet` dispatch. Race condition risk.
- Service intents constructed and fired directly; tightly coupled to `MockLocationService`'s Intent API.

**Fix:** Break `onAction` into domain-grouped handler methods. Remove direct `PreferencesDataSource` injection; expose `getRoamingDefaults()` through `SettingsRepository`. Convert `latestRoamingDefaults` to `StateFlow`. Introduce `MockLocationCommander` interface.

---

### 6. `MapFloatingView.kt` + `MapScreen.kt` — ~150-line MapLibre init duplicated across two files

**Files:** `feature/widget/impl/...MapFloatingView.kt:204–307`, `feature/map/impl/...MapScreen.kt:241–379`

Nearly identical MapLibre style setup: OSM raster source, trace layers, endpoint circle layer, position layer, ephemeral route layers. Diverges only on line widths and one extra source. Also: `MapFloatingView` is 712 lines with three bottom-sheet-sized composables (roaming sheet, tap panel, favorites picker) fully inlined — none extracted.

**Fix:** Extract `fun Style.addStandardLocationLayers(config: MapLayerConfig)` to `:core:map`. Extract `OverlayRoamingSheet`, `TapActionPanel`, `FavoritesOverlayPicker` from `MapFloatingView`.

---

### 7. `FloatingWidgetService.kt` — `startRouteReplay` duplicated from `MapViewModel`

**File:** `feature/widget/impl/...FloatingWidgetService.kt:446–482`

Identical 30-line logic (fetch profile → optionally get return position → teleport to start → build intent → start service) exists in both `FloatingWidgetService` and `MapViewModel`. Also: `Class.forName("com.locationjoystick.app.MainActivity")` at line 660 is a runtime bomb on any rename; business logic assembled inside `setContent {}` (line 248–254).

**Fix:** Extract `StartRouteReplayUseCase`. Replace `Class.forName` with a proper `Intent` or nav callback. Move `isActivityPaused` computation to `WidgetPanelPresenter`.

---

### 8. `WidgetPanelContent.kt` — `FavoritesFloatingView` / `RoutesFloatingView` structurally duplicated

**File:** `feature/widget/impl/...WidgetPanelContent.kt:210–540`

Two 130-line composables with identical outer scaffolding (scrim, rounded card, header row with back/close, `LazyColumn`). Only the domain type and action buttons differ.

**Fix:** Extract `FloatingPickerShell` composable for the shared scaffold.

---

### 9. Triple-duplicated sort logic + triple-duplicated `CheckboxRow`

**Cross-cutting**

`if (newestFirst) list.sortedByDescending { it.createdAt } else list.sortedBy { it.createdAt }` appears in `MapViewModel`, `WidgetPanelPresenter`, and `FloatingWidgetService`. No shared extension exists.

`CheckboxRow` / `WidgetCheckboxRow` / `SettingsCheckboxRow` — three structurally identical composables in three different modules, none in `:core:designsystem`.

**Fix:** Add `fun <T : HasCreatedAt> List<T>.sortedByAge(newestFirst: Boolean)` to `:core:common`. Add `LjCheckboxRow` to `:core:designsystem`.

---

### 10. `AppConstants.kt` — coupling hub inverts dependency direction

**File:** `core/common/...AppConstants.kt`

20 nested objects spanning service action strings, OSRM config, joystick config, widget config, elevation, export, notifications — all in `:core:common`. This forces every module to depend on `:core:common` for constants that semantically belong to specific feature modules. `CooldownConstants` holds two parallel-indexed lists (a latent alignment bug). `MapColorConstants` uses raw `Long` ARGB values that should be `Color` entries in the design system.

**Fix:** Distribute constants to the modules that own them. Replace parallel lists with `List<CooldownTier>`. Move color constants to design system.

---

### 11. `WidgetPanelPresenter.kt` — wrong `RoamingDefaults` + wrong TAG

**File:** `feature/widget/impl/...WidgetPanelPresenter.kt:244–252`

The hardcoded `RoamingDefaults` at line 244 uses `radiusMeters = 1000.0`, `followRoads = false` — contradicting `AppConstants.RoamingConstants.DEFAULT_RADIUS_METERS = 2000.0` and `DEFAULT_FOLLOW_ROADS = true`. **Semantic bug:** the map floating view shows wrong defaults until the flow emits a value. `TAG = "FloatingWidgetService"` means presenter logs are misattributed.

**Fix:** Replace hardcoded defaults with values sourced from `AppConstants.RoamingConstants`. Fix TAG.
